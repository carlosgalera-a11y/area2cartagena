## 3 · App Check enforce

Abrir:
https://console.firebase.google.com/project/docenciacartagenaeste/appcheck

Para cada recurso, pulsar "Enforce":
- **Cloud Functions** → ON
- **Cloud Firestore** → ON
- **Cloud Storage for Firebase** → ON

> ⚠️ Antes de activar enforce en **Firestore**, asegúrate de que los clientes ya envían token App Check. Si no, se rompe la app. El plan es: integrar App Check en frontend en **PR #5 (Parte B.2)** antes de activar enforce — o activar enforce solo en **Functions** ahora (que es lo que consume `askAi`).

Recomendación: activar **Functions enforce** ahora (askAi ya lo exige con `enforceAppCheck: true`). Activar Firestore + Storage enforce **después** de PR #5.

## 4 · TTL policies en Firestore

Sin estas, las colecciones `aiCache` y `rate_limits_ip` crecen indefinidamente. La función ya respeta `expiresAt` en lectura, pero el clean-up automático lo hace Firestore.

Abrir:
https://console.firebase.google.com/project/docenciacartagenaeste/firestore/ttl

Pulsar "Create policy" dos veces:

| Collection ID      | Timestamp field |
|--------------------|-----------------|
| `aiCache`          | `expiresAt`     |
| `rate_limits_ip`   | `expiresAt`     |

Documentación: https://firebase.google.com/docs/firestore/ttl

## 5 · Validar que `askAi` responde

```bash
firebase functions:shell
```

Dentro del shell:
```js
askAi({
  type: 'educational',
  prompt: 'Di OK en una palabra',
  systemPrompt: 'Eres un formador de MFyC',
}, { auth: { uid: 'test-uid' } })
```

Debe devolver `{ provider: 'deepseek', model: 'deepseek-chat', text: 'OK' (o similar), cached: false, ... }`.

> Nota: `functions:shell` no valida App Check, así que el `enforceAppCheck:true` se salta localmente. En producción, la primera llamada real desde el frontend (con token válido) es la prueba de fuego.
